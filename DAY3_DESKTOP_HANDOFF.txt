1. 선택한 아키텍처와 선택 이유
======================================================================

1-1. 기존 웹 애플리케이션 구조

- 패키지 관리자: npm
- 프레임워크: Next.js 15 App Router
- UI: React 18 + TypeScript
- PDF 관련 라이브러리:
  - pdf-lib: PDF 병합, 분할, 페이지 복사, 회전, 다시 저장, 이미지 삽입
  - pdfjs-dist: PDF 페이지 렌더링 및 PDF → 이미지 변환
  - JSZip: 분리된 PDF 및 변환 이미지를 ZIP 파일로 묶기
- 주요 PDF 처리 진입점:
  - lib/client/processPdfJob.ts
  - app/tools/[tool]/tool-workspace.tsx
- PDF 파일은 처리 서버로 업로드되지 않는다.
- File, Blob, ArrayBuffer, Canvas, URL.createObjectURL 같은 브라우저 API로 현재 기기 메모리에서 처리한다.
- 서버 Route Handler는 PDF 처리에 사용되지 않는다.
- 웹 전용 서버 Route Handler:
  - app/api/analytics/route.web.ts: 개인정보를 포함하지 않는 제품 이벤트 기록
  - app/api/health/route.web.ts: 웹 서비스 상태 확인

1-2. 데스크톱 아키텍처

- 선택: Tauri v2
- 선택하지 않은 방식: Electron, 별도 Node.js 서버 번들, 원격 웹사이트만 표시하는 데스크톱 셸
- Tauri를 선택한 이유:
  1. 기존 PDF 기능이 브라우저 안에서 동작하므로 Windows WebView2에서도 같은 코드를 재사용할 수 있다.
  2. PDF 처리용 Node.js 백엔드나 Next.js 서버 런타임을 데스크톱 앱에 포함할 필요가 없다.
  3. Electron보다 설치 파일과 런타임 크기를 작게 유지할 수 있다.
  4. 기존 웹 애플리케이션은 그대로 유지하면서 데스크톱 빌드만 정적으로 분리할 수 있다.
  5. 사용자 파일이나 비밀 환경 변수를 Rust 명령으로 전달할 필요가 없다.

1-3. 웹 빌드와 데스크톱 빌드의 분리 방식

- 일반 웹 빌드:
  - 명령어: npm run build
  - Next.js 서버 Route Handler를 포함한다.
  - 분석 이벤트 POST API와 health API를 유지한다.
- 데스크톱 프런트엔드 빌드:
  - 명령어: npm run desktop:frontend
  - 내부적으로 DESKTOP_BUILD=1 next build --no-lint를 실행한다.
  - next.config.mjs에서 output: "export"가 활성화된다.
  - trailingSlash가 활성화되어 Tauri 로컬 자산 경로에서 내부 탐색이 동작한다.
  - 결과 디렉터리: out
- Tauri 프로덕션 빌드:
  - 명령어: npm run desktop:build
  - Tauri가 먼저 npm run desktop:frontend를 실행한다.
  - 생성된 out 디렉터리를 Windows 앱에 포함한다.

1-4. 웹 전용 Route Handler 분리

- Next.js 정적 내보내기는 POST Route Handler를 지원하지 않는다.
- 그래서 웹 전용 Route 파일을 route.ts가 아니라 route.web.ts로 변경했다.
- 일반 웹 빌드의 pageExtensions에는 web.ts가 포함된다.
- DESKTOP_BUILD=1일 때는 pageExtensions에서 web.ts를 제외한다.
- 이 방식으로 웹 API는 유지하면서 데스크톱 정적 빌드에는 서버 API가 포함되지 않는다.
- 관련 파일:
  - next.config.mjs
  - app/api/analytics/route.web.ts
  - app/api/health/route.web.ts

1-5. 데스크톱 런타임과 인터넷 연결

- PDF 합치기: 로컬 처리, 인터넷 불필요
- PDF 나누기: 로컬 처리, 인터넷 불필요
- 페이지 순서 변경·회전·삭제: 로컬 처리, 인터넷 불필요
- PDF → JPG/PNG: 로컬 처리, 인터넷 불필요
- JPG/PNG → PDF: 로컬 처리, 인터넷 불필요
- PDF 다시 저장: 로컬 처리, 인터넷 불필요
- 설치 파일 다운로드와 향후 업데이트 확인에는 인터넷이 필요할 수 있다.
- 데스크톱 앱에는 Next.js 분석 API가 없으므로 lib/analytics.ts가 Tauri 런타임을 감지하면 서버 전송을 건너뛴다.
- 분석 이벤트 전송 여부는 PDF 처리 성공 여부에 영향을 주지 않는다.

1-6. 데스크톱 보안 경계

- Tauri 창은 번들에 포함된 로컬 out 자산만 로드한다.
- 원격 PDF 처리 서버나 원격 웹 UI를 로드하지 않는다.
- 외부 http, https, mailto 링크는 components/ExternalLinkGuard.tsx가 가로채 시스템 기본 브라우저 또는 이메일 앱에서 연다.
- Tauri opener 플러그인은 opener:allow-open-url 권한만 사용한다.
- Tauri CSP는 기본적으로 self, asset, blob, data 및 IPC에 필요한 범위만 허용한다.
- NEXT_PUBLIC_ 환경 변수는 데스크톱 번들에 공개될 수 있으므로 비밀 값, 토큰, 서명 인증서, API 비밀 키를 넣으면 안 된다.
- Windows 코드 서명 인증서는 프런트엔드 환경 변수가 아닌 CI 또는 안전한 서명 환경에서 관리해야 한다.

6. 공개 배포 전 반드시 남은 작업
======================================================================

6-1. Rust 툴체인 설치

현재 환경 진단:

- Windows: 사용 가능
- WebView2: 설치됨
- Visual Studio Community 2022 / MSVC: 설치됨
- rustc: 없음
- cargo: 없음
- rustup: 없음

권장 설치:

1. https://rustup.rs/에서 rustup-init.exe를 받는다.
2. Windows MSVC 기본 stable 툴체인을 설치한다.
3. 새 PowerShell을 연다.
4. 다음 명령으로 확인한다.

rustup --version
rustc --version
cargo --version

완료 조건:

- 세 명령이 모두 버전을 출력한다.
- rustc host가 x86_64-pc-windows-msvc 계열인지 확인한다.

6-2. Tauri 개발 빌드 실행

명령어:

npm run desktop:dev

- PDF Toolkit 창이 1280 × 820 기본 크기로 열린다.
- 960 × 640보다 작게 줄어들지 않는다.

공통 테스트:

- 지원하지 않는 확장자
- 20MB 초과 개별 파일

failed to run cargo metadata --no-deps --format-version 1: program not found

6-6. 코드 서명과 배포 보안

- 공개 배포용 Windows 코드 서명 인증서를 준비한다.
- 인증서나 비밀번호를 저장소, package.json, NEXT_PUBLIC_ 환경 변수에 넣지 않는다.
- CI secret 또는 별도의 안전한 서명 환경을 사용한다.
- 서명된 실행 파일의 게시자 이름과 서명 유효성을 Windows에서 확인한다.
- 설치 파일에 악성 코드 검사를 수행한다.
- 다음 명령 등으로 SHA-256 체크섬을 만든다.

Get-FileHash -Algorithm SHA256 <installer-path>

- 오프라인 지원은 실제 인터넷 차단 테스트 통과 후에만 명시한다.

6-8. 실제 다운로드 URL 연결

- GitHub Release에 첨부된 설치 파일의 직접 HTTPS URL을 확보한다.
- 웹 배포 환경 변수에 다음 값을 설정한다.

NEXT_PUBLIC_WINDOWS_INSTALLER_URL=https://github.com/<owner>/<repo>/releases/download/v0.1.0/<installer-file>

- 웹앱을 다시 빌드하고 배포한다.
- /download에서 Windows 카드가 Available로 바뀌는지 확인한다.
- 다운로드 버튼이 정확한 GitHub Release 자산을 가리키는지 확인한다.
- 버튼을 클릭해 실제 설치 파일 다운로드가 시작되는지 확인한다.
- 잘못된 URL이나 HTML 페이지가 다운로드되지 않는지 확인한다.
- URL 검증 전에는 환경 변수를 설정하지 않는다.

7. 현재 위험, 제한 및 대응 방안
======================================================================

7-1. Windows 네이티브 빌드 미완료

현황:

- 데스크톱 정적 프런트엔드는 성공했다.
- Tauri Rust 바이너리와 Windows 설치 파일은 생성되지 않았다.
- 원인은 Rust/Cargo/rustup 부재다.

영향:

- 현재 상태를 공개 Windows 지원 또는 Available로 표시하면 안 된다.
- 실제 WebView2에서 PDF 처리 기능이 동작한다고 최종 확정할 수 없다.

대응:

- 6-1부터 6-5까지 완료한다.
- 그전에는 /download의 "Windows build in testing" 상태를 유지한다.

7-2. 실제 데스크톱 회귀 테스트 미완료

현황:

- 코드 구조상 여섯 기능은 로컬 WebView에서 동작하도록 구성됐다.
- 정적 HTML/JS 산출물과 모든 도구 경로는 생성됐다.
- 하지만 Rust 부재로 Tauri 창에서 실제 파일을 선택하고 처리하는 테스트는 실행하지 못했다.

영향:

- WebView2의 파일 선택, Blob 다운로드, PDF.js worker, Canvas 렌더링 또는 CSP 관련 런타임 차이가 남아 있을 수 있다.

대응:

- Tauri dev에서 모든 기능을 실제 샘플로 테스트한다.
- 개발자 콘솔에서 worker-src, asset URL, blob URL 오류를 확인한다.
- 실패가 있으면 CSP 범위를 필요한 출처만 최소한으로 조정한다.

7-3. 암호화·손상 PDF 제한

현황:

- pdf-lib와 pdfjs-dist가 읽지 못하는 암호화 또는 심하게 손상된 PDF가 존재한다.
- PDF 다시 저장 기능은 모든 손상 파일을 복구하는 도구가 아니다.

영향:

- 사용자가 복구 보장으로 오해할 수 있다.

대응:

- 암호화 파일은 암호 보호를 해제하라는 오류를 표시한다.
- 손상 파일은 유효한 PDF인지 확인하라는 오류를 표시한다.
- UI와 변경 내역에서 복구를 보장하지 않는다.

7-4. 메모리 사용량

현황:

- 파일은 서버가 아니라 앱 WebView 메모리에서 처리한다.
- PDF → 이미지 변환은 페이지별 Canvas와 ZIP 데이터를 만들기 때문에 큰 문서에서 메모리를 많이 사용한다.
- 현재 제한은 개별 파일 20MB, 작업 전체 50MB다.

영향:

- 페이지 수, 해상도, 이미지 내용과 기기 RAM에 따라 제한 이내 파일도 실패할 수 있다.

대응:

- 메모리 부족 오류를 친절하게 표시한다.
- 릴리스 테스트에서 저사양 4GB RAM 환경과 큰 샘플 PDF를 확인한다.
- 향후 페이지별 스트리밍 또는 순차 ZIP 기록을 검토한다.
- 검증 없이 파일 한도를 높이지 않는다.

7-5. npm audit의 sharp 고위험 advisory

현황:

- npm audit --omit=dev에서 sharp/libvips 관련 high severity 취약점 2건이 보고됐다.
- 관련 advisory: CVE-2026-33327, CVE-2026-33328, CVE-2026-35590, CVE-2026-35591 묶음
- sharp는 Next.js 의존성 경로에서 들어온다.
- 현재 앱 코드는 next/image를 사용하지 않는다.
- npm audit fix --force는 Next.js 14.2.35로 강제 변경하는 결과를 제안해 적용하지 않았다.

영향:

- 직접 사용하지 않더라도 프로덕션 의존성 감사가 깨끗하지 않다.
- 잘못된 강제 수정은 Next.js 15 앱의 회귀를 만들 수 있다.

대응:

- npm audit fix --force를 그대로 실행하지 않는다.
- Next.js 또는 sharp의 호환 가능한 패치 릴리스를 확인한다.
- 수정 버전으로 별도 브랜치에서 업데이트 후 lint, 타입, 웹 빌드, PDF 기능을 재검증한다.
- 웹 배포에서 이미지 최적화 엔드포인트를 사용하지 않는 현 상태도 확인한다.

7-6. 코드 서명 부재

현황:

- Windows 설치 파일이 아직 없고 코드 서명도 구성하지 않았다.

영향:

- 서명되지 않은 설치 파일은 Windows SmartScreen 경고를 유발할 수 있다.
- 사용자가 앱의 게시자를 신뢰하기 어렵다.

대응:

- 공개 Release 전에 신뢰할 수 있는 인증서로 설치 파일에 서명한다.
- 서명 인증서는 저장소와 데스크톱 번들에 포함하지 않는다.
- 서명된 파일의 해시와 서명을 Release 전에 다시 검증한다.

7-7. 자동 업데이트 미구현

현황:

- 현재는 설치 파일 빌드와 수동 GitHub Release 연결만 준비했다.
- Tauri updater는 구성하지 않았다.

영향:

- 첫 공개 버전 이후 사용자는 새 설치 파일을 직접 받아야 한다.

대응:

- Day 3 범위에서는 자동 업데이트를 추가하지 않는다.
- 첫 설치 파일과 수동 업데이트 흐름을 안정화한 다음 별도 작업으로 Tauri updater, 서명 키 관리, 롤백 정책을 설계한다.

7-8. PWA는 완전한 대체 수단이 아님

현황:

- manifest는 있지만 서비스 워커와 완전한 아이콘 구성이 없다.
- /download에는 PWA 설치 버튼이 없다.

영향:

- 브라우저별 설치 가능 여부가 일관적이지 않다.
- PWA를 Windows 네이티브 앱과 동일하게 안내하면 안 된다.

대응:

- PWA가 필요하면 별도 작업에서 service worker, 캐시 정책, 아이콘, 업데이트 동작, 오프라인 테스트를 구현한다.
- PWA와 Tauri Windows 앱을 UI에서 명확히 구분한다.

7-9. 분석 이벤트 차이

현황:

- 웹앱은 /api/analytics로 최소 제품 이벤트를 보낸다.
- 데스크톱 앱은 정적 번들이므로 서버 이벤트를 보내지 않는다.

영향:

- 웹과 데스크톱 사용량을 동일한 이벤트 데이터로 비교할 수 없다.

대응:

- 현재 개인정보 우선 범위에서는 데스크톱 전송 비활성화를 유지한다.
- 향후 데스크톱 분석이 필요하면 명시적 정책, opt-out, 최소 payload, HTTPS endpoint, 실패 격리 방식을 먼저 설계한다.
- 파일 이름, 파일 내용, 파일 크기, 오류 원문, 개인 식별자를 수집하지 않는다.

7-10. 공개 지원 범위

현황:

- Windows를 목표로 구성했지만 실제 설치 파일 검증 전이다.
- macOS와 Linux 빌드는 존재하지 않는다.

금지되는 표현:

- Windows Available: 실제 검증된 설치 파일 URL이 없을 때 사용 금지
- macOS 지원: 사용 금지
- Linux 지원: 사용 금지
- 완전한 오프라인 지원: 인터넷 차단 상태의 실제 Tauri 테스트 전까지 사용 금지
- 100% 안전, 완전한 보안, 모든 PDF 복구: 사용 금지

허용되는 현재 표현:

- Windows build in testing
- macOS Coming Soon
- Linux Coming Soon
- PDF 파일은 현재 구현상 기기 안에서 처리됨
- 공개 설치 파일은 아직 제공하지 않음
